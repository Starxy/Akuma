使用 Tinc 建立 VPN 网络

Tinc VPN

和 OpenVPN、IPsec、L2TP、WireGuard 一样，Tinc 也是一种可以建立虚拟专用网络的工具。不过相比于以上的工具，Tinc 有以下特点：

  • 非常易于扩展
  • 全 Mesh 网络：存在多公网 ip 节点情况下即便是一个节点断掉也不会影响到其他节点的通信
  • 易于网络管理
  • 全平台支持

在这些特性下，我们可以搭建一个私有网络，可以远程连接 Tinc 内的设备进行办公、在 Tinc 内部署各类服务、通过 Tinc 进行游戏直连等等。

拓扑结构

目前我已经部署使用了半年 Tinc 网络，虽说只有一个公网 ip 节点，但是在某些场景下有一个 VPN 还是会带来极大的方便。

现有的 VPN 网络拓扑结构如下：

  • VPN 网段：10.1.2.0/24
  • 服务端 IP 段：10.1.2.1 - 10.1.2.10
  • 客户端 IP 段：10.1.2.11 - 10.1.2.255

Tinc 本身是不区分服务端和客户端的，每个 tincd 都会默认监听 655 端口上的通信，等待其他 tincd 的链接请求，任何在 vpn 节点内的设备都是平等通信。在这里我们把具有公网 ip 和全部路由表信息的设备称作服务端。

主中心节点为一台 j1800 的服务器，也是目前的唯一公网 ip 节点，其他的在分发完公钥配置完毕后都会尝试连接到该设备，之后该设备会分发密钥和路由表信息来使得节点内的其他设备能够互相可达。

从零开始搭建 Tinc 网络

生命周期

想要合理的规划配置一个 Tinc 虚拟专用网络，我们必须得对 Tinc 的运行周期有所了解，这样才能从全局去考虑配置方向。

如上所说，Tinc 本身是不区分服务端和客户端的，一个 Tinc 守护进程（tincd）启动后，会解析命令行参数读入指定的 tinc.conf 配置文件（该文件称为主配置文件），并尝试连接 ConnectTo 字段指定的其他的 Tinc 守护进程，并且会自动重试连接直到成功。无论是否有 ConnectTo 字段，tincd 启动后会监听其他 tinc 守护进程的连接请求。在未与其他节点建立连接之前，当前 tinc 守护进程可达的节点必须在 hosts 文件夹内有其相关的配置文件（host 配置文件），该文件指定了其他节点的公网地址，虚拟网内的地址，公钥等信息。在与其他节点建立通信之后，会进行公钥交换，可以与更多节点建立通讯。
